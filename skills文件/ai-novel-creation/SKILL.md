# AI Novel Creation Skill

用于长篇小说创作、续写、扩写和质量修复。核心目标不是“写够字”，而是让每一章都发生足够多的具体事件，并让这些事件形成清晰的因果链，让人物关系、局势、线索或能力至少发生一种不可逆变化。

## 0. 最高优先级

1. 正文有效字数不得低于3000字。正文只包含章节正文，不包含标题、元数据、质量检查表、说明文字。
2. 禁止自报“约3000字”。必须统计实际正文非空白字符数；低于3000时继续扩写，不能写完成标志。
3. 先写“章节蓝图”，再写正文。没有蓝图不得直接起草章节。
4. 每章至少包含4个完整场景，且至少3个场景必须出现明确阻力、对话、选择或行动后果。
5. 任何只概述结果的句子都要展开成场景。比如“众人开始重建”“敌人全部投降”“他想明白了”不是情节，只是摘要；必须写出谁做了什么、遇到什么阻碍、如何选择、造成什么后果。
6. 如果单次输出无法写满3000字，分多轮或分段追加到同一章节文件；在达标前不得标记完成。
7. 每章必须有“上章问题 -> 本章目标 -> 中段阻力/反转 -> 代价 -> 新问题”的结构链。任何场景不能只靠“然后”连接，必须能说明上一场景怎样触发下一场景。

## 1. 项目文件结构

新建小说时创建：

```text
{小说名}/
├── project.md
├── foreshadowing.md
└── chapters/ 或 {N}卷_{卷名}/
    └── {NN}章_{章节标题}.md
```

续写时必须先读取：

- `project.md`：世界观、人物、已完成章节、当前进度、下一章方向。
- `foreshadowing.md`：已埋、推进、待回收伏笔。
- 最近1-3章正文：确认语气、节奏、人物状态和未解决冲突。

如果项目已有自己的目录格式，沿用现有格式，不强行迁移。

## 2. project.md 必备内容

`project.md` 至少维护这些信息：

```markdown
# {小说名}

## 作品设定
- 类型：
- 时间与地点：
- 核心体系：
- 主要矛盾：

## 人物表
| 角色 | 身份 | 欲望/目标 | 恐惧/弱点 | 说话风格 | 当前状态 |
|------|------|-----------|-----------|----------|----------|

## 信息状态表
| 信息/秘密/误解 | 已知角色 | 未知角色 | 证据来源 | 可能误解 | 后续影响 |
|---------------|----------|----------|----------|----------|----------|

## 章节规划
| 章节 | 标题 | 章节功能 | 本章冲突 | 结尾钩子 |
|------|------|----------|----------|----------|

## 已完成章节
| 章节 | 标题 | 正文实测字数 | 有效事件数 | 开场模式 | 状态变化 | 结尾钩子 |
|------|------|--------------|------------|----------|----------|----------|

## 当前进度
- 当前章节：
- 主角状态：
- 主要关系变化：
- 未解决冲突：
- 信息差与误解：
- 下一章必须处理：
```

## 3. foreshadowing.md 必备内容

```markdown
# 伏笔追踪表

| foreshadow_id | 内容 | 状态 plant/advance/resolve | 章节 | 影响 | 预计回收 |
|---------------|------|----------------------------|------|------|----------|

## 待处理线索
- 

## 已回收伏笔
- 
```

伏笔必须参与情节推进。只在结尾列出但正文没有体现，视为无效。

## 4. 续写前读取顺序

每次写新章节前按顺序执行：

1. 读取 `project.md`，确认当前章节、人物状态、下一章必须处理的问题。
2. 读取 `foreshadowing.md`，选择本章要 plant/advance/resolve 的伏笔。
3. 读取最近1-3章正文，提取上一章结尾钩子、上一次开场模式、人物情绪余波。
4. 提取上一章遗留的“问题、代价、信息差、未完成承诺”，不能只提取剧情摘要。
5. 写出本章“章节蓝图”。
6. 蓝图通过后才写正文。

## 5. 章节蓝图

正文前必须先生成并遵守以下蓝图。蓝图可以写在工作笔记中，最终章节文件可不保留完整蓝图，但写作过程必须使用。

```markdown
## 本章蓝图
- 章节号/标题：
- 章节类型：快节奏/中节奏/慢节奏
- 故事弧段：exposition/rising_action/falling_action/climax/resolution
- 开场模式：A动作/事件入场、B场景承压入场、C悬念/异常入场
- 上章钩子：
- 本章核心问题：
- 本章结尾钩子：
- 上章遗留问题：
- 本章必须解决的问题：
- 本章不能解决但必须推进的问题：
- 主角主动目标：
- 反作用力来源：
- 中段反转/失败：
- 结尾新压力：
- 本章必须造成的变化：
  1. 局势变化：
  2. 人物关系变化：
  3. 信息/伏笔变化：

## 人物状态变化表
| 角色 | 开始时想要什么 | 本章被迫面对什么 | 做出的选择 | 付出的代价 | 结束时状态 |
|------|----------------|------------------|------------|------------|------------|

## 节奏曲线
- 开场压力：低/中/高
- 中段压力变化：升高/误导/短暂缓和/反转
- 结尾压力：必须高于开场，或留下更具体的问题
- 本章读者期待：获得答案/看到冲突/等待反击/担心代价

## 场景卡
| 场景 | 预计字数 | 地点/时间 | 在场人物 | 场景目标 | 阻力/冲突 | 新信息 | 选择/行动 | 后果 | 下一场景触发原因 | 未解决问题 |
|------|----------|-----------|----------|----------|-----------|--------|-----------|------|------------------|------------|
| 1 | 500-800 | | | | | | | | | |
| 2 | 700-1000 | | | | | | | | | |
| 3 | 700-1000 | | | | | | | | | |
| 4 | 600-900 | | | | | | | | | |
| 5 可选 | 400-800 | | | | | | | | | |
```

蓝图最低要求：

- 至少4张场景卡。
- 至少2个外部事件：袭击、谈判、交易、试炼、调查发现、资源争夺、救援、背叛、追捕等。
- 至少1个内部变化：人物改变态度、承认恐惧、做出艰难选择、关系推进或破裂。
- 至少1个伏笔操作：plant/advance/resolve。
- 场景之间必须有因果链，不能只是“然后又发生一件事”。
- 每个非结尾场景都必须写出“下一场景触发原因”；如果写不出，说明该场景与主线脱节，需要合并、改写或删除。

## 6. 正文结构与字数

正文有效字数：3000-5000字。建议分配：

| 部分 | 字数 | 功能 |
|------|------|------|
| 开场 | 400-700 | 承接上章，抛出本章核心问题 |
| 发展一 | 800-1100 | 第一个行动或调查，出现阻力 |
| 发展二 | 800-1100 | 冲突升级，人物通过对话/行动改变局面 |
| 转折 | 600-900 | 新信息、失败、代价或艰难决定 |
| 结尾 | 400-700 | 本章结果落地，并留下下一章动力 |

快节奏章节也不能低于3000字。慢节奏章节必须用“关系变化、信息揭示、价值冲突”填充，不能用天气、感慨、背景说明凑字。

结构顺滑要求：

- 开场必须承接上一章的钩子或代价，不能像新故事重新开始。
- 发展一必须由开场问题触发，不能随机换任务。
- 发展二必须让阻力升级，不能只是重复同类行动。
- 转折必须改变主角的判断、资源、关系或风险，不能只补设定。
- 结尾钩子必须由本章行动自然导致，不能凭空塞入新危机。

## 7. 场景写作标准

每个场景必须具备：

- 场景目标：人物当下想得到什么。
- 阻力：另一个人、环境、规则、资源不足、误判、时间压力或内心禁忌。
- 行动：人物采取具体行为，而不是只思考。
- 对话或互动：除极少数独处场景外，每个场景至少有一段有目的的对话/互动。
- 结果：场景结束时，信息、关系、位置、资源、伤势、承诺或危险至少改变一项。
- 触发：除最后一个场景外，每个场景的结果必须触发下一场景的行动、冲突、误解或追索。

写法要求：

- 用“展示”替代“总结”。不要写“他们忙了一下午”，要写出一个有阻力的片段：谁负责、哪里坏了、缺什么、谁提出反对、主角如何取舍。
- 对话必须带目的：试探、隐瞒、逼问、安抚、谈判、威胁、交换信息。
- 心理描写必须由具体事件触发，并影响下一步选择。
- 环境描写必须服务于行动、危险、情绪或线索；纯装饰删掉。

## 8. 反水文硬规则

以下内容不能计入有效推进，出现后必须替换或展开：

- 空泛总结：一切都变好了、众人充满希望、他终于明白责任、敌人毫无还手之力。
- 跳过过程：开始重建、完成训练、敌人投降、双方达成协议。
- 重复系统面板但没有新代价、新限制或新选择。
- 连续两段以上无人行动、无人对话、无新信息的环境或心理描写。
- 角色说话只表达感谢、鼓励、担心，但不改变决策或关系。
- 反派或困难被一句话解决。
- 一句话跨越关键过程：凡是胜负、谈判、训练突破、关系破裂、重大发现、能力升级、阵营转向，都必须写出至少一个完整场景。

替换方法：

1. 给摘要句增加具体阻力。
2. 让至少两名角色对解决方式产生分歧。
3. 写出主角的选择代价。
4. 让结果留下一个新问题。

## 9. 开场模式轮换

相邻章节不得使用同一开场模式。

| 模式 | 用法 |
|------|------|
| A 动作/事件入场 | 直接从行动、对话、危险或任务现场开始 |
| B 场景承压入场 | 用环境细节承载压力，并迅速引出人物选择 |
| C 悬念/异常入场 | 先抛出异常、疑问、坏消息或不合理现象 |

开场模式必须写入章节末尾元数据，并更新 `project.md`。

## 10. 人物一致性

写人物前检查：

- 角色当前目标是什么。
- 角色最害怕失去什么。
- 角色知道哪些信息，不知道哪些信息。
- 角色说话风格是否独特。
- 本章结束时角色状态有什么变化。
- 角色本章的选择是否来自既有欲望、恐惧、误解或压力，而不是作者需要他这么做。

人物不能为了推动剧情突然变聪明、变蠢、变善良或变邪恶。需要转变时，必须给出事件触发、内心阻力和外部代价。

重要角色在本章发生变化时，必须能回答：

| 角色 | 变化前 | 触发事件 | 内心阻力 | 外部代价 | 变化后 |
|------|--------|----------|----------|----------|--------|

## 11. 世界观与能力系统

涉及系统、魔法、科技、资源、组织规则时：

- 写清限制、代价、冷却、消耗或风险。
- 新能力必须制造新问题，不能只让主角无成本碾压。
- 资源变化要进入后续情节：获得、消耗、损坏、被争夺、被误用。
- 规则一旦建立，后续章节必须遵守；如果要突破规则，必须付出更高代价、暴露隐藏限制或引出新的敌对反应。

## 12. 写作完成后的真实验收

章节正文写完后，必须执行以下验收。不能只写勾选符号。

### A. 字数验收

统计正文非空白字符数。优先使用随 skill 提供的校验脚本：

```powershell
.\skills文件\ai-novel-creation\scripts\validate_chapter.ps1 -Path "章节文件路径.md"
```

也可以手动用 PowerShell 统计：

```powershell
$text = Get-Content -LiteralPath "章节文件路径.md" -Raw -Encoding UTF8
$body = ($text -split '---')[0]
($body -replace '\s','').Length
```

如果无法运行命令，也必须手动估算并保守处理；只要有疑问就继续扩写。低于3000字不得完成。

### B. 故事推进验收

```markdown
## 故事推进验收
| 项目 | 具体内容 |
|------|----------|
| 本章开始状态 | |
| 本章结束状态 | |
| 事件1 | |
| 事件2 | |
| 事件3 | |
| 事件4 | |
| 人物关系/状态变化 | |
| 伏笔操作 | |
| 主角付出的代价 | |
| 下一章钩子 | |
```

任何一项写不出具体内容，说明正文需要补写。

### C. 场景密度验收

```markdown
| 场景 | 是否有目标 | 是否有阻力 | 是否有对话/互动 | 是否有后果 | 是否触发下一场 |
|------|------------|------------|----------------|------------|----------------|
| 1 | | | | | |
| 2 | | | | | |
| 3 | | | | | |
| 4 | | | | | |
```

少于4个有效场景，或有两个以上场景没有后果，必须扩写。

### D. 因果与逻辑验收

```markdown
## 因果链验收
| 项目 | 具体内容 |
|------|----------|
| 上章遗留问题如何进入本章 | |
| 场景1如何触发场景2 | |
| 场景2如何触发场景3 | |
| 场景3如何触发场景4 | |
| 主角主动推动局面的行动 | |
| 反作用力造成的真实损失 | |
| 本章代价如何影响下一章 | |
| 结尾钩子为什么不是凭空出现 | |
```

任何一项只能写出“然后”“与此同时”“突然”，说明因果链不足，必须改写相邻场景。

## 13. 章节末尾元数据

只有通过真实验收后，才能在章节末尾写：

```markdown
---

**本章开场模式：A/B/C**
**正文实测字数：{实际数字}字**
**有效场景数：{数字}**
**有效事件数：{数字}**
**故事推进验证：通过**
**因果链验证：通过**
**主角付出代价：{具体代价}**
**下一章钩子：{具体钩子}**
```

禁止写“约3000字”“预计字数”“达标”但没有实际统计数字。

## 14. 保存与更新

完成章节后：

1. 保存章节正文到项目既有章节目录。
2. 更新 `project.md` 的已完成章节表：正文实测字数、有效事件数、开场模式、状态变化、结尾钩子。
3. 更新 `project.md` 的信息状态表：本章新增信息、误解变化、哪些角色知道或仍不知道。
4. 更新 `foreshadowing.md`：本章新增、推进或回收的伏笔。
5. 如果章节低于3000字、未通过故事推进验收或未通过因果链验收，不得更新为完成。

## 15. 修复低质量章节流程

当用户要求修复已有章节，按以下步骤：

1. 统计现有正文实际字数。
2. 列出缺失：有效场景数、有效事件数、人物变化、伏笔、结尾钩子。
3. 先补章节蓝图和因果链，保留原有可用情节。
4. 把摘要句扩写为场景，优先增加阻力、对话、选择代价、后果和下一场触发原因。
5. 重写或扩写到正文不少于3000字。
6. 重新验收并更新 `project.md` / `foreshadowing.md`。

## 16. 常见失败与修正

| 失败 | 修正 |
|------|------|
| 后期章节越来越短 | 强制先写4-5个场景卡，每个场景有目标、阻力、后果 |
| 描写很多但剧情没动 | 每800字至少出现一次行动、对话、信息揭示或选择 |
| 场景像清单一样并列 | 为每个非结尾场景补“下一场触发原因”，不能补就合并或删除 |
| 人物突然改变立场 | 补触发事件、内心阻力、外部代价，必要时提前一章埋转变压力 |
| 结尾钩子像硬塞 | 让钩子由本章行动、误判、代价或伏笔自然引出 |
| 系统流变成面板播报 | 每次面板变化都绑定代价、限制、敌人反应或下一步任务 |
| 决战一笔带过 | 分成侦察、接敌、受挫、反制、代价、收尾至少6个动作节点 |
| 角色只会感谢/担心 | 让角色提出要求、隐瞒信息、质疑主角、改变计划 |
| 自称达标但字数不足 | 使用真实字数统计，正文不足就继续写，不许写完成标志 |
